Git Workflow

About

The core idea behind the Feature Branch Workflow is that all feature development should take place in a dedicated branch instead of the master branch.

★ ★ ★

This encapsulation makes it easy for multiple developers to work on a particular feature without disturbing the main codebase.

★ ★ ★

It also means the master branch will never contain broken code, which is a huge advantage for continuous integration environments.

★ ★ ★

Getting started

Start by cloning a repository to your local machine or pull the latest changes to your current one .

Features

  • Develop new features for upcoming releases
  • Typically exist in developers repos only

★ ★ ★

Start a new feature

Development of new features starting from the 'develop' branch.

Start developing a new feature with

git checkout develop
git checkout -b MYFEATURE

This action creates a new branch based on 'develop' and switches to it

Finish up a feature

Finish the development of a feature.

  • Switch back to 'develop' branch
  • Merge MYFEATURE into 'develop'
  • Remove the feature branch
git checkout develop
git merge MYFEATURE
git branch -d MYFEATURE

Publish a feature


Publish a feature to the remote server so it can be used by other users.

git push -u origin develop

Getting a published feature

Get a feature published by another user.

git checkout develop
git pull

Make a release

  • Support preparation of a new production release
  • Allow for minor bug fixes and preparing meta-data for a release

★ ★ ★

Start a release

To start a release, create a release branch from develop branch

★ ★ ★

Modify Readme File ,the new package names in cfg.xml

★ ★ ★

And follow the folders name convention

git checkout -b rel_num

★ ★ ★

It's wise to publish the release branch after creating it to allow release commits by other developers. Do it similar to creating a new branch:

git push -u origin rel_num

Finish up a release

  • Merges the release branch back into 'master'
  • Tags the release with its name
git checkout master
git merge release_no
git tag -a v25 -m 'Production version 25'
git push -u origin master

  • Back-merges the release into 'develop'
  • Removes the release branch
git checkout develop
git merge release_no
git push -u origin develop
git branch -d release_no

Don't forget to push your tags withgit push --tags
